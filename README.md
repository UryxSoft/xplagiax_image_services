## What is xplagiax Images Search Services?

**XplagiaX ISS** is a production-grade microservice that combines three AI capabilities into a single REST API:

- **AI vs Human Detection** — classifies whether an image was generated by AI or created by a human, using a fine-tuned SigLIP model (2024)
- **Visual Similarity Search** — finds visually similar images using CLIP embeddings and approximate nearest-neighbor search via Qdrant
- **Plagiarism Analysis** — detects copied, modified, cropped, recoloured, or watermarked versions of images with configurable similarity thresholds
- **Patent & Reverse Image Search** — searches the web and patent databases using external APIs with intelligent rotation and circuit breaking

Built to run at scale: async ML inference, distributed object storage via SeaweedFS, vector indexing via Qdrant, and full observability out of the box.

### Why this stack?

| Component | Choice | Reason |
|-----------|--------|--------|
| **Embeddings** | CLIP ViT-B/32 | Best open-source visual similarity model |
| **AI Detection** | SigLIP fine-tuned | State-of-the-art AI vs Human classifier (2024) |
| **Vector DB** | Qdrant + HNSW | Sub-100ms ANN search, payload indexes, cosine similarity |
| **Object Storage** | SeaweedFS | Purpose-built for billions of small files, no metadata DB needed |
| **Cache / Queue** | Redis | Atomic counters, embedding cache, async job queue |
| **API Server** | Gunicorn + gevent | Non-blocking I/O, single worker keeps one model copy in RAM |

## Features

- **Idempotent indexing** — same image always produces the same UUID (SHA-256 based). Re-uploading never creates duplicates.
- **Async ML inference** — Flask responds in `<100ms`. Heavy CLIP/SigLIP inference runs in a dedicated RQ worker process.
- **Graceful degradation** — Redis down? Service keeps running. SigLIP failed to load? CLIP search still works. Every failure is isolated.
- **Zero vector poisoning protection** — corrupt images are rejected before touching Qdrant. No silent bad data.
- **Path traversal prevention** — all filenames and group IDs are sanitised with allowlists before filesystem or storage operations.
- **Structured JSON logging** — every log line includes `request_id`, `endpoint`, `group_id`, and `duration_ms` for full traceability.
- **Prometheus metrics** — 20+ metrics including P95 latency, model confidence distribution, cache hit rates, and collection size.
- **SeaweedFS storage** — images stored in distributed object storage. Switching from local to SeaweedFS requires only one env var change.

## API Overview

All endpoints require `X-API-Key: <your-key>` header (or `Authorization: Bearer <key>`).

### Images

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/api/v1/images` | Upload and index an image |
| `POST` | `/api/v1/images/batch` | Upload up to 20 images at once |
| `GET` | `/api/v1/images/<id>` | Retrieve image binary |
| `GET` | `/api/v1/images/<id>/url` | Get direct storage URL |
| `DELETE` | `/api/v1/images/<id>` | Delete image from Qdrant + SeaweedFS |

### Search & Analysis

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/api/v1/search/similar` | Find visually similar images |
| `POST` | `/api/v1/search/plagiarism` | Full plagiarism report with alert level |
| `POST` | `/api/v1/search/ai-detection` | Classify image as AI-generated or human |

### Patents & Reverse Search

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/api/v1/patents/search/image` | Find patents related to an image |
| `POST` | `/api/v1/patents/search/text` | Find patents by keyword |
| `GET` | `/api/v1/patents/<id>` | Get full patent details |
| `POST` | `/api/v1/patents/reverse-image` | Reverse image search on the web |
| `GET` | `/api/v1/patents/usage` | API quota usage status |

### Reverse Image Search (lightweight, no local ML)

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/api/v1/reverse-image-search` | Provider-agnostic reverse image search with cost-saving early-stop |
| `POST` | `/api/v1/reverse-image-search/batch` | Same, for multiple images in one request (`files` multipart list and/or `image_urls` JSON list, capped at `REVERSE_SEARCH_MAX_BATCH_SIZE`). Images are independent and processed concurrently; each result includes a `source` field. Returns `207` with `{"count", "results": [...]}`. |

Independent from the CLIP/SigLIP/Qdrant stack above: this endpoint only
orchestrates external reverse-image-search APIs (Google Vision Web
Detection, Serper Lens, optional experimental Mungfali) in priority order,
and returns exactly:

```json
{
  "found": true,
  "website": "Wikipedia",
  "url": "https://es.wikipedia.org/...",
  "similarity": 99.0,
  "provider": "google_vision",
  "elapsed_ms": 412
}
```

The chain stops the instant one provider clears its own configured
`*_STOP_THRESHOLD` — a confident match on provider 1 never spends quota on
providers 2/3. Identical images (by SHA-256) are served from cache without
calling any provider at all. See `.env.example`
(`REVERSE_SEARCH_*`, `GOOGLE_VISION_*`, `SERPER_*`, `MUNGFALI_*`) and
`app/reverse_search/`.

This module can also run as a **standalone, torch-free microservice** for
deployments that only need reverse image search — see
`requirements-reverse-search.txt` and `docker/Dockerfile.reverse-search` /
`docker/docker-compose.reverse-search.yml`. It shares zero import-time
dependency on `app.models`, `app.storage.vector_repository`, or
`app.workers`, so torch/transformers/qdrant-client are never loaded. The
standalone deploy also serves `/api/v1/patents/*` (SerpApi/Zenserp — see
below) since that feature has the same zero-heavy-ML footprint; it does
**not** serve `/api/v1/images` or `/api/v1/search/*`, which need the
Qdrant/CLIP/SigLIP stack and only exist in the monolith.

## Similarity Thresholds

| Score | Match Type | Interpretation |
|-------|-----------|----------------|
| ≥ 0.97 | `EXACT_COPY` | Same image, possibly recompressed or resized |
| 0.90 – 0.97 | `MODIFIED_COPY` | Edited: crop, color change, watermark, filter |
| 0.80 – 0.90 | `HIGH_SIMILARITY` | Visually related, possible derivative work |
| 0.70 – 0.80 | `MODERATE_SIMILARITY` | Some visual relation — manual review recommended |
| < 0.70 | No match | Unrelated images |

## Quick Start

### Prerequisites

- Docker 24+
- 8 GB RAM minimum (12 GB recommended)
- 20 GB free disk space

### 1 — Clone the repository

```bash
git clone https://github.com/youruser/xplagiax.git
cd xplagiax
```

### 2 — Create the Docker network

```bash
docker network create xplagiax-net
```

### 3 — Start infrastructure services

```bash
# Redis
docker run -d --name redis-server --network xplagiax-net \
  -p 6379:6379 -v redis_data:/data --restart unless-stopped \
  redis:7.2-alpine redis-server --maxmemory 2gb --maxmemory-policy allkeys-lru

# Qdrant
docker run -d --name qdrant --network xplagiax-net \
  -p 6333:6333 -p 6334:6334 -v qdrant_data:/qdrant/storage --restart unless-stopped \
  qdrant/qdrant:v1.10.1

# SeaweedFS Master
docker run -d --name seaweedfs-master --network xplagiax-net \
  -p 9333:9333 -v seaweedfs_data:/data --restart unless-stopped \
  chrislusf/seaweedfs:latest master \
  -ip=seaweedfs-master -ip.bind=0.0.0.0 -port=9333 -mdir=/data

# SeaweedFS Volume  (wait 5 seconds after master)
docker run -d --name seaweedfs-volume --network xplagiax-net \
  -p 8080:8080 -v seaweedfs_data:/data --restart unless-stopped \
  chrislusf/seaweedfs:latest volume \
  -mserver=seaweedfs-master:9333 -ip=seaweedfs-volume \
  -ip.bind=0.0.0.0 -port=8080 -dir=/data -max=0

# SeaweedFS Filer  (wait 5 seconds after volume)
docker run -d --name seaweedfs-filer --network xplagiax-net \
  -p 8888:8888 -v seaweedfs_filer_data:/data --restart unless-stopped \
  chrislusf/seaweedfs:latest filer \
  -master=seaweedfs-master:9333 -ip=seaweedfs-filer \
  -ip.bind=0.0.0.0 -port=8888
```

### 4 — Configure environment

```bash
cp .env.example .env
nano .env
```


### 5 — Build the image

```bash

```

### 6 — Start the application

```bash

```


## Security

- API key authentication via `X-API-Key` or `Authorization: Bearer`
- Rate limiting: 30 req/min, 500 req/hour per IP (configurable)
- Hard image size limit: 20 MB (enforced at Nginx + WSGI + validation layers)
- Magic byte validation — file type verified by content, not extension
- Path traversal protection on all filesystem and storage operations
- Deterministic UUIDs from SHA-256 content hash — prevents enumeration
- Internal paths and stack traces never exposed in API responses
- Confirmation token required for destructive admin operations

## ¿Qué es xplagiax?

**xplagiax** es un microservicio de producción que combina tres capacidades de IA en una sola REST API:

- **Detección IA vs Humano** — clasifica si una imagen fue generada por IA o creada por un humano, usando un modelo SigLIP fine-tuned (2024)
- **Búsqueda por Similitud Visual** — encuentra imágenes visualmente similares usando embeddings CLIP y búsqueda de vecinos más cercanos aproximados via Qdrant
- **Análisis de Plagio** — detecta copias, versiones modificadas, recortadas, recoloreadas o con marca de agua, con umbrales de similitud configurables
- **Búsqueda de Patentes e Imágenes Inversas** — busca en la web y bases de datos de patentes usando APIs externas con rotación inteligente y circuit breakers

Diseñado para escalar: inferencia ML asíncrona, almacenamiento distribuido via SeaweedFS, indexación vectorial via Qdrant, y observabilidad completa desde el primer día.

## Características

- **Indexación idempotente** — la misma imagen siempre produce el mismo UUID (basado en SHA-256). Re-subir nunca crea duplicados.
- **Inferencia ML asíncrona** — Flask responde en `<100ms`. El pesado proceso de inferencia CLIP/SigLIP corre en un worker RQ dedicado.
- **Degradación elegante** — ¿Redis caído? El servicio sigue funcionando. ¿SigLIP falló? La búsqueda CLIP sigue operando. Cada fallo está aislado.
- **Protección contra vector cero** — las imágenes corruptas son rechazadas antes de tocar Qdrant. Sin datos silenciosamente malos.
- **Prevención de path traversal** — todos los nombres de archivo y group IDs son sanitizados con allowlists.
- **Logs JSON estructurados** — cada línea incluye `request_id`, `endpoint`, `group_id` y `duration_ms` para trazabilidad completa.
- **Métricas Prometheus** — más de 20 métricas incluyendo latencia P95, distribución de confianza del modelo y tasas de cache hit.
- **Almacenamiento SeaweedFS** — imágenes en almacenamiento objeto distribuido. Cambiar de local a SeaweedFS: un solo cambio de variable de entorno.

| `POST` | `/api/v1/patents/search/text` | Buscar patentes por texto |

### Búsqueda Inversa de Imágenes (ligera, sin IA local)

| Método | Endpoint | Descripción |
|--------|----------|-------------|
| `POST` | `/api/v1/reverse-image-search` | Búsqueda inversa orquestando proveedores externos con Early Stop |
| `POST` | `/api/v1/reverse-image-search/batch` | Igual, para varias imágenes en un solo request (`files` multipart o `image_urls` JSON, tope `REVERSE_SEARCH_MAX_BATCH_SIZE`). Cada imagen es independiente y se procesa concurrentemente; cada resultado incluye `source`. Devuelve `207` con `{"count", "results": [...]}`. |

Independiente del stack CLIP/SigLIP/Qdrant: este endpoint solo orquesta APIs
externas de reverse image search (Google Vision Web Detection, Serper Lens,
Mungfali experimental) en orden de prioridad y devuelve exactamente:

```json
{
  "found": true,
  "website": "Wikipedia",
  "url": "https://es.wikipedia.org/...",
  "similarity": 99.0,
  "provider": "google_vision",
  "elapsed_ms": 412
}
```

La cadena se detiene apenas un proveedor supera su propio
`*_STOP_THRESHOLD` configurado — una coincidencia confiable en el proveedor
1 nunca consume cuota de los proveedores 2/3. Imágenes idénticas (por
SHA-256) se sirven desde cache sin llamar a ningún proveedor. Ver
`.env.example` (`REVERSE_SEARCH_*`, `GOOGLE_VISION_*`, `SERPER_*`,
`MUNGFALI_*`) y `app/reverse_search/`.

Este módulo también puede correr como **microservicio standalone sin
torch**, para despliegues que solo necesitan reverse image search — ver
`requirements-reverse-search.txt` y `docker/Dockerfile.reverse-search` /
`docker/docker-compose.reverse-search.yml`. El standalone también sirve
`/api/v1/patents/*` (SerpApi/Zenserp), porque esa función tiene el mismo
perfil sin IA local; **no** sirve `/api/v1/images` ni `/api/v1/search/*`,
que necesitan el stack Qdrant/CLIP/SigLIP y solo existen en el monolito.

## Umbrales de Similitud

| Score | Tipo | Interpretación |
|-------|------|----------------|
| ≥ 0.97 | `EXACT_COPY` | Misma imagen, posiblemente recomprimida o redimensionada |
| 0.90 – 0.97 | `MODIFIED_COPY` | Editada: recorte, cambio de color, marca de agua, filtro |
| 0.80 – 0.90 | `HIGH_SIMILARITY` | Muy relacionadas, posible obra derivada |
| 0.70 – 0.80 | `MODERATE_SIMILARITY` | Alguna relación visual — revisión manual recomendada |
| < 0.70 | Sin coincidencia | Imágenes no relacionadas |

## 2) Subir / indexar imagen — `POST /api/v1/images`

Fuentes admitidas: `file` (multipart), `image_url` (http/https), `image_path`
(local, solo si `ALLOW_LOCAL_IMAGE_PATH=true`). Campos opcionales: `group_id`,
`page`, `run_ai_detection` (default true), `extra_*` (metadatos). Devuelve **202**
(modo async) o **200** (modo degradado síncrono).

```bash
# Archivo local (multipart)

  -H "X-API-Key: $API_KEY" \
  -F "file=@/ruta/imagen.jpg" \
  -F "group_id=proyecto_x" \
  -F "run_ai_detection=true" \
  -F "extra_origen=campaña-2026"
```

```python
# Archivo local
with open("imagen.jpg", "rb") as f:
```

## 🤖 Modelo de detección de IA y método de reverse search

- **Detección IA**: `Ateeqq/ai-vs-human-image-detector` — SigLIP2 ViT-B/16 (~86M params),
  **INT8 ≈ 150–250 MB RAM** en CPU; ~99.23% in-distribution. Configurable con
  `SIGLIP_MODEL_ID` (+ `AI_LABEL_MAP` para modelos con etiquetas no estándar).
- **Reverse search**: **Google Lens** vía SerpApi (`REVERSE_IMAGE_ENGINE=google_lens`),
  con fallback a `google_reverse_image` y Zenserp; resultados normalizados + veredicto.
- **Honestidad**: ambos son **probabilísticos** — no afirman certeza. Detalle en
  `docs/MODELS_AND_REVERSE_SEARCH.md`.
